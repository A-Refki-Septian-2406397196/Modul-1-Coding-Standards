## Reflection 4 module 3

1. Saya menerapkan prinsip SOLID untuk membuat struktur kode lebih rapi dan mudah dikembangkan. Pertama, saya menerapkan SRP dengan memisahkan tanggung jawab tiap layer: ProductController dan CarController saya pisahkan (tanpa inheritance) agar controller hanya fokus pada routing HTTP, binding data, dan memilih view/redirect; aturan use-case seperti pembuatan ID saat id masih null dan logika update saya taruh di layer service; sedangkan repository saya sederhanakan menjadi penyimpanan in-memory murni untuk operasi dasar seperti save/find/findAll/delete, termasuk merapikan kontrak method (misalnya mengembalikan List daripada Iterator) dan menghapus kode yang tidak dipakai. Kedua, saya menerapkan OCP secara minimal pada fitur Car dengan memisahkan kebijakan pembuatan ID ke interface CarIdGenerator, sehingga jika format ID berubah saya cukup menambah implementasi baru tanpa mengubah CarServiceImpl. Ketiga, saya menerapkan LSP dengan menghapus CarController extends ProductController karena sebelumnya CarController tidak benar-benar bisa menggantikan ProductController (bahkan sampai super(null)), lalu menjadikannya controller mandiri sesuai kontraknya. Keempat, saya menerapkan ISP dengan memecah service Car menjadi dua interface yang lebih kecil, yaitu interface untuk membaca data (findAll, findById) dan interface untuk manipulasi data (create, update, delete), supaya class yang memakai service hanya bergantung pada fungsi yang dibutuhkan. Terakhir, saya menerapkan DIP dengan memastikan controller bergantung pada abstraksi (interface seperti CarQueryService dan CarCommandService) bukan pada implementasi konkret, sehingga implementasi service bisa diganti atau dikembangkan tanpa perlu mengubah controller.

2. Menurut saya, keuntungan utama menerapkan prinsip SOLID adalah membuat proyek lebih mudah dirawat dan dikembangkan karena setiap komponen punya peran yang jelas. Setelah saya memisahkan ProductController dan CarController (SRP) tanpa inheritance, perubahan pada fitur Car tidak lagi “menyentuh” alur Product, sehingga risiko bug silang berkurang dan debugging jadi lebih cepat karena saya tahu titik perubahan ada di file yang relevan saja. Untuk penambahan fitur, dampaknya juga terasa: pada fitur Car saya menerapkan OCP minimal dengan CarIdGenerator, jadi ketika format ID perlu diubah (misalnya ditambah prefix atau mengikuti pola tertentu), saya cukup menambahkan implementasi generator baru tanpa mengubah CarServiceImpl. Saya juga menerapkan LSP dengan menghapus CarController extends ProductController, karena sebelumnya CarController tidak benar-benar bisa menggantikan ProductController (bahkan sampai super(null)), dan ini berpotensi menimbulkan error; setelah dipisah, perilaku tiap controller lebih konsisten dan aman. Selain itu, ISP membantu saya memecah service Car menjadi interface read (CarQueryService) dan manipulasi data (CarCommandService), sehingga komponen yang hanya butuh read tidak ikut bergantung pada operasi write—ini membuat dependensi lebih kecil dan perubahan lebih terkontrol. Terakhir, dengan DIP controller bergantung pada interface, bukan implementasi (CarServiceImpl), sehingga kalau suatu saat penyimpanan data Car diganti dari in-memory ke database, saya bisa mengganti implementasi service/repository tanpa perlu mengubah controller. Jadi, secara keseluruhan SOLID membuat penambahan fitur lebih aman, perubahan lebih terlokalisasi, dan proses tracking error jauh lebih mudah karena struktur kode tidak saling “terikat” secara berlebihan.

3. Jika saya tidak menerapkan SOLID, proyek saya cenderung menjadi sulit dirawat karena banyak komponen punya tanggung jawab yang bercampur dan saling bergantung, sehingga perubahan kecil bisa menimbulkan efek samping ke bagian lain. Contohnya, sebelum SRP saya sempat menaruh CarController dan ProductController dalam satu file bahkan memakai inheritance (CarController extends ProductController); akibatnya ketika saya mengubah fitur Car, saya berisiko ikut mengganggu alur Product, dan bahkan pernah muncul masalah seperti pemanggilan super(null) yang membuat kontrak ProductController tidak terpenuhi dan bisa memicu error yang sulit dilacak. Tanpa OCP, aturan seperti pembuatan ID akan tertanam langsung di CarServiceImpl, sehingga ketika format ID berubah saya harus mengedit class yang sama berulang kali dan berpotensi menambah banyak if/else yang membuat kode makin kompleks. Tanpa ISP, service yang besar memaksa class lain bergantung pada banyak method yang sebenarnya tidak dipakai, sehingga perubahan pada salah satu method bisa memaksa refactor di banyak tempat walaupun fitur yang berubah kecil. Tanpa DIP, controller akan langsung bergantung pada implementasi seperti CarServiceImpl, sehingga ketika saya ingin mengganti implementasi (misalnya dari in-memory ke database), perubahan akan merambat sampai ke controller dan membuat risiko bug lebih tinggi. Intinya, tanpa SOLID, kode menjadi lebih rapuh, sulit diuji, sulit diperluas, dan ketika error terjadi saya akan lebih lama mencari sumber masalah karena dependensi dan tanggung jawabnya tidak terpisah dengan jelas.
